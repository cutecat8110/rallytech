# 設計系統規則總綱

最後更新：2026-04-02

## 文件目的

- 定義本專案的 design system foundations、semantic-first 原則、Nuxt UI bridge 邏輯與 styling ownership matrix。
- 讓設計、前端與 AI agent 在同一套治理模型下協作，不再把 `main.css` 當作所有 section 樣式的總倉。

## 適用角色

- 設計：確認品牌色彩、字體、surface 與 layout 的固定邊界。
- 前端：確認 token、semantic CSS variables、Nuxt UI bridge 與 local styling 的責任分工。
- AI agent：在切版與視覺調整時，先遵守這份 foundations 主檔，再決定哪些規則屬於全域、區塊或單點例外。

## 1. Design Governance 的單一來源

執行期與文件的真相來源順序如下：

1. `app/assets/css/main.css`
   - CSS-first theme primitives
   - Nuxt UI semantic CSS variables
   - shared layout / surface / media primitives
2. `app/app.config.ts`
   - Nuxt UI global theme bridge
   - `ui.colors`
   - 全域 component theme（`button`、`input`、`select`、`textarea`、`tabs` 等）
3. `docs/design/01 / 02 / 03`
   - foundations、Nuxt UI implementation rules、Vision-to-Code grammar
4. 其他展示頁與設計文件

規則：

- `main.css` 與 `app/app.config.ts` 是 runtime 真相。
- `docs/design/` 是治理與協作真相。
- 若三者不一致，先修正一致再合併。

## 2. Semantic-First 原則

本專案採 Nuxt UI 的 semantic-first 設計系統：

- `Primary`：品牌主 CTA、主操作、關鍵互動
- `Secondary`：次要操作、深色骨架、補充動作
- `Neutral`：文字、邊框、背景、disabled 與一般 UI 狀態
- `Info / Success / Warning / Error`：狀態色，不兼任品牌主色

實作順序固定為：

1. 先在 `@theme` 定義 primitives（字體、色票、breakpoints、radius）
2. 再在 `:root` / `.dark` 定義 `--ui-*` semantic CSS variables
3. 再在 `app/app.config.ts` 透過 `ui.colors` 映射 `primary / secondary / neutral`
4. 最後由元件、頁面與區塊使用 semantic utilities 與 Nuxt UI component props

禁止事項：

- 不在頁面散落 hex
- 不直接把 reference 的色票硬貼到 template
- 不把單一 section 的幾何與 crop 規則提升成全域 token

## 3. Styling Ownership Matrix

### Layer 1：`app/assets/css/main.css`

負責：

- `@theme` primitives：`--font-*`、`--color-*`、`--breakpoint-*`、`--radius-*`
- Nuxt UI semantic CSS variables：`--ui-primary`、`--ui-secondary`、`--ui-text`、`--ui-bg`、`--ui-border`、`--ui-radius`、`--ui-container`、`--ui-header-height`
- shared layout shells：例如 `page-sys-shell`、`page-sys-shell--wide`、`content-sys-rail`
- shared semantic utilities 與 shared media primitives

不負責：

- 單一 section 的 spacing、crop、overlay、connector geometry
- 單頁專屬的 block grammar

### Layer 2：`app/app.config.ts`

負責：

- `ui.colors` 的 semantic mapping
- Nuxt UI 元件的全域 theme bridge
- `slots`、`variants`、`compoundVariants`、`defaultVariants` 的全域覆寫

規則：

- 只放真正跨頁一致的 component theme
- 不把 route 條件樣式或 section 特例塞進全域 config
- 覆寫結構必須對齊元件原本的 theme shape

### Layer 3：`<UTheme>` / `:ui`

負責：

- 區塊級或單頁級的 Nuxt UI 變體
- 特定 section 的 button / tabs / input 等 component 語氣

規則：

- 只在 block-level 需要與全域不同時使用
- 優先 `<UTheme>`，單點才用 `:ui`

### Layer 4：component local class / local `<style scoped>`

負責：

- 單一 section 專屬視覺
- spacing、crop、overlay、connector breakout、特殊 layout grammar

規則：

- 只用在單一區塊的樣式，預設留在元件本地
- 不再把首頁所有 section grammar 長期堆在 `main.css`

## 4. Promotion Rule

同一視覺規則若跨兩個以上 section 重複，才有資格提升：

1. 先評估是否可用 Nuxt UI semantic colors / global theme 解決
2. 再評估是否可提升為 shared primitive 或 utility
3. 若只用在單一區塊，維持 component local

簡單判準：

- token、semantic vars、shell、shared primitive：進 `main.css`
- global component theme：進 `app/app.config.ts`
- block-level Nuxt UI variation：進 `<UTheme>` / `:ui`
- 單一區塊 geometry / spacing / crop：留在元件本地

## 5. Foundations

### 字體

- 全站字體：`IBM Plex Sans TC`
- 中、英、數同字族，不維持雙軌字體策略
- 英文層級以同一家族內的 `weight / tracking / token role` 建立差異，不直接複製 reference 的雙字體對比
- 字重分工固定如下：
  - `700`：`display-*`、CJK major heading、重要 CTA
  - `600`：`title-*`、`type-sys-nav`、`type-sys-kicker`、mixed-script title、English accent heading
  - `500`：`label-*`、按鈕與表單標籤
  - `400`：`body-*` 與主要敘述文字
- 中文不使用 uppercase 規則
- 英文 uppercase tracking 控制在 `0.08em ~ 0.12em`
- `type-sys-nav` 預設使用 `13px / 130% / 600 / 0.04em`
- `type-sys-kicker` 預設使用 `13px / 120% / 600 / 0.10em`

### 色彩

- `Primary`：科技青綠，承接主 CTA 與品牌互動
- `Secondary`：石墨深色骨架，承接深色底與次要互動
- `Neutral`：文字、背景、邊框、disabled
- `Accent`：低頻點綴，不承接品牌主操作

### Radius

- 基準半徑：`--ui-radius = 4px`
- `rounded-*` 使用 Nuxt UI / Tailwind 系統比例，不自建任意半徑尺度

### Layout

- `--ui-container` 對應 boxed content shell
- `page-sys-shell / page-sys-shell--wide / content-sys-rail` 為 shared shell grammar
- section 專屬 geometry 不進 container token

### Hero hierarchy

- `immersive hero` 僅首頁使用，承接品牌氛圍、主敘事與首屏級 CTA。
- `page hero` 為首頁以外的標準 hero，優先讓頁面身份與正文入口快速可見。
- Hero height scale 屬於 shared primitive，應由共用 hero shell 承接，不直接提升為全域 layout token。
- `immersive hero` 與 `page hero` 可共用 shell 結構，但不共享 overlay 配方、crop、grid、glow 與單頁 copy rhythm。
- 內頁 hero 的一致性以視覺節奏為主：同高度、同留白、同內容重心；若屬入口頁，可保留輕量差異，但不得脫離 `page hero` 基準。
- 內頁 `page hero` 預設採單一 H1-first grammar，不在 hero 內堆疊副標、說明或多組裝飾訊號；額外資訊應回到正文區塊承接。
- 內頁 `page hero` 的 H1 預設採單語顯示：繁中 locale 只顯示中文，英文 locale 只顯示英文，不在同一個 H1 內混入對譯文案。

## 6. Ownership 範例

- `shared media-frame` base 可留在 `main.css`
- `Mission` 專屬 light / square / trapezoid / focus / overlay 變體應回元件本地
- `Services / OTE` 的 background surface 若只被單一 section 使用，應回元件本地
- `Header / Hero` 這類單頁 section-only grammar，若無第二使用點，也應留在 component local `<style scoped>`
- 若多個內頁共用同一組 hero 高度與 layering，應優先抽成 shared primitive，例如 `PageHeroShell`，而不是把各頁 hero selector 堆回 `main.css`

## 7. 與 `02 / 03` 的邊界

- `01` 只負責 foundations、semantic-first、ownership matrix 與 promotion rule。
- `02` 只負責 Nuxt UI implementation order、`app/app.config.ts` / `<UTheme>` / `:ui` / local style 的使用方式。
- `03` 只負責 Vision-to-Code grammar、Architecture Pack / Implementation Contract 欄位與 component-driven 拆解順序。

## 8. 驗證要求

任何 design system 與 ownership 調整，至少需完成：

1. `npm run lint`
2. `npm run typecheck`
3. 首頁與 `/design-system` 的視覺抽查
4. 若首頁 pilot 有遷移，額外做 reference/local 截圖回歸
