# 整站文案節奏與區塊分工規範

最後更新：2026-04-22

## 文件目的

- 定義 Rally Technology 官網的 page job、section job 與整站 copy system。
- 讓 `source -> en -> zh-tw -> runtime` 的改稿有共同準繩，不再出現同頁重複、各區塊各說各話的情況。
- 作為首頁、about、services overview、service detail、one-touch、contact、FAQ 後續改稿與審稿的共同依據。

## 適用範圍

- `i18n/locales/en.ts`
- `i18n/locales/zh-tw.ts`
- `i18n/locales/services/en-detail.ts`
- `i18n/locales/services/zh-tw-detail.ts`
- `docs/references/en/*`
- `docs/references/zh-tw/*`

## 手動文稿鎖定

- `docs/rewrite/文案.md` 是 2026-04-22 公司提供的服務文案手動修正來源。
- 本次手動稿保留既有 8 項服務 IA 與路由；`IT建置服務` 僅映射為 services overview 的整體能力脈絡，不新增服務內頁。
- 已由手動稿映射至 public i18n 的 `en` / `zh-tw` 內容，除非使用者提供新的手動修正文稿或明確要求覆寫，不得被 AI 主動重寫、翻新、同步成 generated copy，或以 reference copy 覆蓋。
- `source` locale 與 `reference-pages.generated.ts` 維持 evidence / QA 層，不因本次 runtime manual copy mapping 自動更新。

## 1. 整站 page job

- 首頁：定位 + 信任 + 能力地圖 + 下一步
- About：公司角色 + 工作方式 + 品牌基礎
- Services overview：服務分類與差異
- Service detail：承接內容、做法、交付價值
- One-touch：跨系統整合層
- Contact：快速轉換
- FAQ：輔助說明，不承擔品牌主敘事

## 2. 訊息白名單與出現位置

### 可重複使用但需控管的核心訊息

- `2014 成立`
- `automation control and monitoring systems`
- `專業 / 迅速 / 團隊 / 誠信`
- `ISO 9001 / ISO 45001`
- `WinCC OA Partner`
- `再生能源 / 離岸風電 / 化工 / 製藥 / EPC / 工業`
- `SCADA / PI / Historian / PLC / DCS`

### 區塊分配規則

- `2014`：
  - 每頁最多一次。
  - 優先出現在首頁 about、about intro、footer/company meta。
  - 不放首頁 hero。
- `automation control and monitoring`：
  - 可作為首頁 hero 的定位。
  - about 與 services overview 只承接，不重複原句。
- `專業 / 迅速 / 團隊 / 誠信`：
  - 不以口號清單鋪滿全站。
  - 優先出現在首頁 mission 與 about intro 第 2 段，轉成工作方式。
- `ISO / WinCC OA Partner`：
  - 優先出現在首頁 about point、about intro 第 2 段、後續 proof 區塊。
  - 不進 hero 主標。
- `SCADA / PI / Historian / PLC / DCS`：
  - 由 services overview、service detail、one-touch 主導。
  - 不進首頁 hero。

## 3. 首頁 section job

### Hero

- 只回答「雷力是做什麼的」與「第一層價值是什麼」。
- 結構固定：
  - 一句定位
  - 一句價值
  - 一個 CTA
- 不塞年份、認證、partner、平台名單、產業名單。

### About

- 首頁上的 about 是 `Why Rally / 信任摘要`，不是第二個 hero。
- 只放：
  - 公司角色
  - 品牌 / 交付價值
  - proof 線索
- 三個 points 各自回答不同問題，不可同義重寫。

### Services

- 只做能力地圖與差異分類。
- 不重講品牌態度與交付哲學。
- 卡片 description 用一致句型：
  - `服務對象 / 承接內容 / 交付用途`

### Process

- 只回答「怎麼合作」。
- 以步驟與節點為主，不寫品牌宣言。

### One-touch

- 只回答「哪些跨系統整合會一起被處理」。
- 不是第九個服務，也不是泛化的整合宣言。
- 語言集中在：
  - `SCADA`
  - `alarm workflow`
  - `Historian`
  - `utilities`
  - `cross-system flow`

### Mission

- 作為首頁的品牌態度收束區。
- 承接 `專業 / 迅速 / 團隊 / 誠信`，但轉成工作方式與合作感受。
- 不重講 process。

## 4. 其他頁型 section job

### About

- intro 第 1 段：雷力是誰、做什麼、服務哪些專案情境。
- intro 第 2 段：怎麼做事、為何可信。
- capability tiles：只放技術能力，不混品牌口號。
- process：維持短步驟，不寫成第二個 mission。

### Services overview

- hero：服務總覽定位與常見產業情境。
- body intro：可作為 8 張服務卡前的「分類導讀」，協助讀者依系統層級理解服務入口。
- body intro 不可直接複製 SEO description，也不可重複 hero 主張；應說明如何閱讀服務範圍。
- overview card：只回答「這項服務主要處理什麼工作」，不展開內頁首屏的工作邊界說明。
- 8 張卡片要明確分出：
  - 畫面
  - 現場
  - 控制程式
  - 即時資料平台
  - 歷史資料
  - 網路
  - 遠端資料
  - 告警流程

### Service detail

- `heroFocusItems`：快速看懂範圍
- `intro`：工作邊界 + 現場條件 + 交付判斷，不重複 overview card description。
- `media-feature`：說明該服務在實務上容易失控的原因，以及需要先整理的結構；不可把 intro 句型再說一次。
- `proof-strip`：可信訊號，不是第二段自我介紹
- `capability-list`：做法
- `summary / closing-note`：只談何時需要提早納入，以及晚處理會讓哪些邊界更難收斂。

### One-touch

- 頁面角色是跨系統整合層。
- 不與 services overview 重複列服務清單。
- hero 後維持直接進入 module stack，由各 module 承擔跨系統整合說明；預設不新增 body intro，避免重複列出 SCADA、告警、Historian 與公用系統。

### Contact

- hero、intro、form help 都回到「快速、清楚、專業」。
- intro 使用 `SharedSectionIntro` grammar 管理 kicker / title；兩段 paragraph 可保留在下方 body，不再使用左側 bar marker。
- 表單尚未串接實際送出流程時，submit 後必須同時提供 toast 與表單內 inline feedback；inline feedback 需可見且可被螢幕閱讀器讀到，不得只依賴浮動 toast。
- 只回答：
  - 什麼情況適合聯絡
  - 我們會怎麼回應
  - 下一步是什麼

### FAQ

- 若仍是 placeholder，就明確維持 support page 角色。
- 不承擔品牌主敘事。
- 不複製 contact page 的說服邏輯。

## 5. 重複限制

- 同一頁不可在 3 個以上區塊重複同一組主張。
- `規劃 / 現場 / 交接` 這類完整句型每頁最多一次。
- `整合、交付、支援` 不可連續以同義句重寫，必須換成更具體的頁面責任。
- about 與首頁不能用同一組 opening sentence。
- services overview 不得用 service detail 的 summary 句型倒貼回卡片。
- service detail 首屏不得直接複製 overview card description；應改寫成該服務的工作邊界、現場條件或交付判斷。

## 6. proof 使用層級

### 第一層：可直接放進主文案

- 公司成立年份
- 認證與 partner
- 常見產業情境
- 可辨識的平台與技術名詞

### 第二層：只適合放進 proof 區塊或案例層

- 具名客戶
- 具名專案
- 比較級說法
- 成效數字

## 7. 語言與術語規則

### English

- corporate-ready
- direct verbs first
- no agency tone
- no inflated slogans

### zh-tw

- 台灣 B2B 官網語氣
- 以短句與穩定主詞為優先
- 避免中英混寫拉長句子
- 保留必要術語英文，其餘以中文為主

### 術語一致性

- `SCADA / HMI`
- `PLC / DCS`
- `PI Server`
- `Historian / Historians`
- `industrial network`
- `alarm workflow`
- `remote monitoring and data`

若同一術語在 overview 與 detail 之間需要中文化，必須保持可辨識映射，不可每頁改一種叫法。

## 8. 排版語法與文字 primitive 分工

本專案的文字一致性以「語法一致」為原則，不要求所有 section 共享同一個版型。

- `SharedSectionIntro`：只管理 section-level 的 `kicker / title / description`，用於首頁 section、About intro、Services body intro、Contact intro 與 Process heading。
- `SharedContentHeader`：管理 section 內部的 module / block / item / closing heading，可承接 eyebrow、subtitle 與單段 description；不得反向取代 `SharedSectionIntro`。
- `SharedTextStack`：管理多段 paragraph 的 spacing、line-height、tone 與 lead-first 規則；About intro、Contact intro、service detail intro / summary / closing 這類段落應優先使用它。
- `SharedTextTileGrid`：管理 title + description / paragraphs 的文字格線，用於 proof-strip、card-grid 這類小型文字 tile；不吸收服務卡 navigation、process steps 或整張互動卡片。

排版提升規則：

- section shell、媒體裁切、深淺底與特殊幾何保留在 local component。
- 若只是單一 section 的視覺張力，不提升成 shared primitive。
- 若同一種大中小標 / paragraph / tile grammar 在公開頁面重複兩處以上，優先提升成 shared text primitive。
- Footer closing CTA 可使用 `SharedContentHeader scale="closing"`，但 Footer band layout、media seam 與 CTA 位置仍屬 Footer local。

## 9. 實作順序

1. 先確認是否存在手動文稿鎖定；若存在，先遵守手動稿 scope 與禁止覆寫規則。
2. 先更新本文件
3. 再改 `i18n/locales/*`
4. 若任務包含 reference 對照交付，再同步 `docs/references/en|zh-tw/*`
5. 只有 reference docs 有同步更新時，才執行 `npm run generate:i18n-reference-pages`

## 10. 驗收問題

- 這個區塊有沒有回答跟前一個區塊不同的問題？
- 這一句是不是 page job 需要的訊息，而不是只是換句話再說一次？
- 讀者能不能在 5 秒內理解這個區塊存在的理由？
- 這個 claim 有沒有足夠 proof，或至少有合理的可信線索？
- 中英節奏是否一致，但又沒有落入直譯？
