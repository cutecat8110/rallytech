# 協作與交付規範

最後更新：2026-03-27

## 文件目的

- 收斂協作流程、提交規範與工具基線。
- 提供可直接執行的交付檢查清單。

## 1. 協作流程（最小必做）

1. 先對齊目標、範圍與成功條件
2. 依 `AGENTS.md` 的 Skill 規範選用最小必要 skills
3. 依 `design/` 規範實作並完成驗證
4. 回報需包含：變更摘要、驗證結果、風險與後續建議

## 2. Commit 與 PR 規範

格式：

```text
<type>(<scope>): <subject>
```

規則：

- `type` 英文、`subject` 中文
- Breaking change 使用 `type(scope)!: subject` 並附 `BREAKING CHANGE: ...`
- PR title 與 commit 主旨採同格式
- 合併策略採 `Atomic + Squash`

提交前檢查順序：

1. `pnpm exec prettier --check .`
2. `pnpm lint`
3. `pnpm typecheck`

任一失敗即中止提交，不使用 `--no-verify` 跳過。

## 3. 工具基線（MCP）

核心 MCP：

- `chrome-devtools`（附著 `http://127.0.0.1:9222`）
- `github`
- `figma`

補強策略：

- 頁數過大或需批次抽取時，再加 crawler 類 MCP
- 需正式設計審核報告時，再加規範審核類 skill

## 4. 技能基線（Skills）

核心 skills：

- `uiux-pm-collaboration`
- `frontend-design`
- `nuxt`
- `nuxt-ui`
- `vue`
- `pnpm`
- `document-writer`

原則：

- 先用現有能力完成第一輪，不預設擴充工具鏈
- 只有在明確痛點出現後，才新增技能或 MCP
