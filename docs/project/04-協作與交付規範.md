# 協作與交付規範

最後更新：2026-03-31

## 文件目的

- 收斂協作流程、提交規範、工具基線與官網重建工作流。
- 提供可直接執行的交付檢查清單，避免流程散落在多份文件中。

## 適用角色

- 設計、前端、PM：確認這一輪要怎麼協作、驗證與交付。
- AI agent：作為 `project` 內工具、skills、QA 與 commit/PR 規範的唯一來源。
- Reviewer：確認提交格式與交付檢查是否完整。

## 1. 協作流程（最小必做）

1. 先對齊目標、範圍與成功條件。
2. 依 `AGENTS.md` 的 Skill 規範選用最小必要 skills。
3. 先以參考站、`references`、設計系統與現有程式碼建立共同真相。
4. 依 `design/` 規範實作並完成驗證。
5. 回報需包含：變更摘要、驗證結果、Skill 使用情況、風險與後續建議。

## 2. 工具基線（MCP）

### 核心 MCP

- `chrome-devtools`
  - 官網重建、`references` 製作與視覺核對的主真相來源。
  - 使用基準：`desktop viewport + full-page scroll + 實際互動後可見內容`。
- `github`
  - PR、diff、review 與 repo 檢索。
- `figma`
  - 設計稿、Code Connect、設計資產與元件映射。

### 擴充 MCP

- `playwright`
  - 用途：可重複的瀏覽、滾動、點擊、accordion 展開、hover、tab、form 與 responsive 驗證。
  - 角色：補強 `chrome-devtools`，不取代它。
  - 啟用條件：只要任務需要可重複操作、互動測試或跨 viewport 驗證，就列入標準工具。

### 圖片 sourcing 層

圖片來源不直接靠人工另存，優先順序固定為：

1. `Unsplash API`
2. `Pexels API`
3. `Pixabay API`

固定輸出欄位：

- 搜尋關鍵字
- 圖片預覽或頁面來源
- photographer / source
- license / use note
- download URL
- local filename suggestion

必要環境變數：

- `UNSPLASH_ACCESS_KEY`
- `PEXELS_API_KEY`
- `PIXABAY_API_KEY`

建議優先使用 `~/.codex/skills/asset-sourcing-licensing/scripts/fetch-stock-candidates.mjs` 產出候選清單與來源欄位。

## 3. 技能基線（Skills）

### 核心 skills

- `uiux-pm-collaboration`
- `frontend-design`
- `motion`
- `nuxt`
- `nuxt-ui`
- `vue`
- `nuxt-seo`
- `pnpm`
- `document-writer`

### 自訂 skills

- `b2b-website-copy-strategy`
  - 負責 `source -> en -> zh-tw` 的 B2B 文案策略、頁面訊息層級、CTA 與信任訊號。
- `visual-qa-review`
  - 負責參考站、設計方向與 Nuxt 實作的資深視覺與互動 QA。
- `asset-sourcing-licensing`
  - 負責開源圖片來源選擇、授權備註、下載記錄與替代圖規則。

原則：

- 先用核心基線完成第一輪，不預設全自動化。
- 新增工具必須對應明確痛點，不為了看起來完整而增加維護成本。
- 前端 / 版面 / 互動任務，仍以 `uiux-pm-collaboration` 作為第一個收斂 skill。

## 4. 官網重建標準流程

1. `chrome-devtools` 逐頁做真實畫面校對，建立 `references` 主真相。
2. `playwright` 補足滾動、展開、hover、tab、form、mobile / desktop 的可重複驗證。
3. `uiux-pm-collaboration + frontend-design + motion` 收斂 IA、視覺方向與互動節奏。
4. `b2b-website-copy-strategy` 產出 `source -> en -> zh-tw`。
5. `asset-sourcing-licensing` 搜尋、下載並記錄 demo 圖片。
6. `nuxt + nuxt-ui + vue + nuxt-seo` 落地實作。
7. `visual-qa-review` 做最終設計與互動審查。

## 5. 驗證與 QA

提交前最少檢查：

1. `pnpm exec prettier --check .`
2. `pnpm lint`
3. `pnpm typecheck`

視覺與互動檢查：

- 以瀏覽器實查為主，不只靠程式碼推測。
- 首頁、服務總覽、代表性服務頁、聯絡頁至少抽查桌機與手機版。
- 有 scroll reveal、accordion、tab、form、hover 時，需驗證互動後狀態。

## 6. Commit / PR 規範

本檔即 `project` 內 commit / PR 規範的唯一來源。

格式：

```text
<type>(<scope>): <subject>
```

規則：

- `type` 使用英文，`subject` 使用中文。
- Breaking change 使用 `type(scope)!: subject`，且 body / footer 含 `BREAKING CHANGE: ...`。
- PR title 與 commit 主旨採同格式。
- 合併策略採 `Atomic + Squash`。

## 7. 維運原則與邊界

- `chrome-devtools` 為長期主基線，優先維持穩定可用。
- `playwright` 直接配置在本機 Codex，作為標準互動驗證層。
- 不採用需付費的 crawler 或 browser automation 服務作為官網重建正式基線。
- 自訂 skills 放在 `~/.codex/skills/`，跟 repo 文件一起演進，但不把 repo 內規範依賴到 skill 內容本身。
