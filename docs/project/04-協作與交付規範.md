# 協作與交付規範

最後更新：2026-04-02

## 文件目的

- 定義本機可用能力、workflow 分層、驗證順序與交付門檻。
- 作為本專案 skills / MCP / capability gate 的唯一 baseline matrix。

## 適用角色

- PM、設計、前端：確認這一輪可依賴哪些工具與技能，以及缺什麼時該如何降級。
- AI agent：作為本機 baseline、workflow layer、驗證與提交規範的唯一詳細來源。
- Reviewer：確認 claim level、evidence、交付與 capability 聲明是否一致。

## 1. 協作流程（最小必做）

1. 先對齊目標、範圍與成功條件。
2. 若任務屬於 reference clone / screenshot-to-code / URL clone，先依 `docs/project/05` 完成 artifact gate。
3. 依 `AGENTS.md` 的規則選用最小必要 skills。
4. 依 `docs/design/01 / 02 / 03` 的 owner 邊界實作。
5. 完成驗證並回報變更摘要、驗證結果、skill 使用情況、claim level 與待決策項。

## 2. Current Local Baseline

### 2.1 本機 MCP config 真相

依 `~/.codex/config.toml`，本機目前已配置：

- `chrome-devtools`
- `playwright`
- `context7`
- `fetch`
- `memory`

目前未配置：

- `github`
- `figma`

補充：

- `memory` 在本機是 enabled，但 workflow 上屬 optional layer。
- `github` 不在本機 config 內；若某個 session 或外部 connector 提供它，視為額外的 review / trace integration，而不是當前 baseline。

### 2.2 本機 feature 真相

依 `~/.codex/config.toml`，本機目前：

- `js_repl = true`
- `multi_agent = true`

解讀：

- `js_repl = true` 代表 workstation 已具備互動式 browser / app debug 的準備條件，但實際是否可用，仍要看當前 session 是否真的提供對應工具。
- `multi_agent = true` 代表可進行代理分工，但不代表每輪都應啟用。

### 2.3 本機 skill inventory

目前已安裝的主要 skills 可分為：

- Core build / frontend：`uiux-pm-collaboration`、`frontend-skill`、`nuxt-ui`、`nuxt`、`vue`、`visual-qa-review`
- Clone / asset / copy：`reference-clone-architect`、`asset-sourcing-licensing`、`b2b-website-copy-strategy`
- Docs / strategy / SEO / CRO：`document-writer`、`site-architecture`、`page-cro`、`form-cro`、`copy-editing`、`schema-markup`、`seo-audit`、`analytics-tracking`、`product-marketing-context`、`nuxt-seo`
- Tooling / implementation：`vite`、`vitest`、`vueuse`、`reka-ui`
- Conditional / system：`playwright-interactive`、`imagegen`

補充：

- `playwright-interactive` 屬於本機已安裝 capability，但只有在當前 session 真的有互動式 browser 工具時才算 active。
- `imagegen` 屬於第二階段品牌變體能力，不列為 fidelity phase 的預設依賴。

## 3. Recommended Workflow Layers

### 3.1 Core layer

預設核心層：

- MCP：`chrome-devtools`、`playwright`、`context7`、`fetch`
- Skills：`uiux-pm-collaboration`、`frontend-skill`、`nuxt-ui`、`nuxt`、`vue`、`visual-qa-review`

用途：

- `chrome-devtools`：browser truth capture、computed style、DOM 結構校對
- `playwright`：repeatable interaction QA、multi-viewport capture
- `context7`：Nuxt / Nuxt UI / Tailwind / Vue 文件真相
- `fetch`：HTML / CSS / content 抓取與比對

### 3.2 Optional layer

可選層：

- MCP：`memory`
- Skills：`document-writer`、`site-architecture`、`page-cro`、`form-cro`、`copy-editing`、`schema-markup`、`seo-audit`、`analytics-tracking`、`product-marketing-context`、`nuxt-seo`

用途：

- `memory`：跨 session 保存決策、例外規則與進度摘要
- Optional skills：用於第二層優化、文件整理、SEO、CRO 與策略補強，不預設每輪都啟用

### 3.3 Conditional / session-provided layer

條件層：

- MCP / integrations：`github`、`figma`
- Skills：`playwright-interactive`、`imagegen`

規則：

- `github`：當前本機未配置；若 session 或 connector 提供，可用於 PR、diff、review trace。
- `figma`：非 clone Phase 1 必要，僅在 design mapping、moodboard 或設計協作任務啟用。
- `playwright-interactive`：只有 session 真正提供互動式 browser 能力時才列為 active。
- `imagegen`：只在 fidelity pass 後進入 Phase 2 品牌變體。

## 4. Capability Gate 與 claim labels

### Capability gate

- 缺少 `chrome-devtools`、`playwright`、`context7`、`fetch` 任一核心能力：標記 `degraded-fidelity`
- 缺少 `github` review / diff layer：只有當任務明確需要 PR trace 或 review 協作時，才標記 `degraded-review-trace`
- 缺少 image generation：不阻塞 fidelity pass；若任務已進入品牌變體階段，標記 `degraded-variant`
- `memory`、`figma` 缺失不阻塞 Phase 1 fidelity

### 標準 claim labels

- `same.new-class-ready`
- `degraded-fidelity`
- `degraded-review-trace`
- `degraded-variant`

## 5. Skills 使用原則（Lean core + optional layer）

- 先用 core layer 完成 truth capture、Nuxt UI 落地與 fidelity pass。
- Optional layer 用於第二層優化，不把所有 skills 視為同等重要。
- Conditional skills / tools 必須先確認當前 session 能力，再決定是否列入 active。
- AI 生圖與品牌圖像變體，不得先於 fidelity artifact 與 layout truth。

AI image workflow 的唯一詳細來源：[`./05-Reference-Clone工作流與降級規範.md`](./05-Reference-Clone工作流與降級規範.md)

## 7. 驗證與 QA

提交前最少檢查：

1. `npm run format:check`
2. `npm run lint`
3. `npm run guard:ui-buttons`
4. `npm run typecheck`

clone / fidelity 任務建議補充：

- `npm run workflow:clone:preflight`
- Playwright 或 browser captures
- 首頁與 `/design-system` 抽查

## 8. Commit / PR 規範

格式：

```text
<type>(<scope>): <subject>
```

規則：

- `type` 使用英文，`subject` 使用中文。
- Breaking change 使用 `type(scope)!: subject`，且 body / footer 含 `BREAKING CHANGE: ...`
- PR title 與 commit 主旨採同格式。
- 合併策略採 `Atomic + Squash`。

## 9. 維運原則與邊界

- `~/.codex/config.toml` 是本機 MCP / feature baseline 的單一來源。
- 啟用新 MCP 或 feature 後，需重啟 Codex 再宣稱 capability 已生效。
- `AGENTS.md` 只負責協作 SOP，不再承擔 design system 細則與 clone artifact 細節。
- `docs/design/01 / 02 / 03` 是 design governance、Nuxt UI implementation 與 artifact grammar 真相。
- `docs/project/05` 是 clone artifact、downgrade 與 AI image workflow 真相。
