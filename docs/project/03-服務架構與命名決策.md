# 服務架構與命名決策

最後更新：2026-03-31

## 文件目的

- 收斂服務主清單與命名策略，避免跨檔重複定義。
- 提供導覽短名、正式頁名與雙語對齊規則，作為前台與 references 的共同基準。

## 適用角色

- 管理、業務：快速掌握第一輪會如何對外呈現服務。
- 設計、前端、PM：在選單、卡片、H1 與 SEO 命名上使用同一套名稱。
- AI agent：避免在不同頁面或不同語言中自行發明服務名稱。

## 1. 決策摘要

- 第一輪服務架構承接參考站 8 項服務，不刪減核心語意。
- `一站式整合` 作為第一層獨立頁面，與 8 項服務並行，不吸收其主命名。
- 前台可先維持平鋪承接，3 個主域分組保留為第二輪優化選項。

## 3. 命名策略

命名原則：

- 導覽易查找優先，其次才是完整語義。
- 保留關鍵專有名詞，不為白話化而失真。
- 中文與英文穩定對齊，避免跨語系結構失配。

命名使用規則：

- 主選單、首頁卡片、服務總覽卡片：使用 `導覽短名`。
- 頁面 H1、SEO、正式對照：使用 `正式中文名` / `正式英文名`。
- `zh-tw` H1 建議採 `正式中文名｜正式英文名`。
- `en` 與 `source` 仍保留英文區塊命名，不把中文名混入原文基準層。

## 4. 第二輪優化方向（可選）

3 個技術主域（資訊架構優化用）：

- `監控與控制`
- `資料平台`
- `儀電與網路`

用途：

- 首頁與服務總覽可先顯示主域，再展開 8 項細部服務。
- 主選單可在第二輪改成 mega menu，按主域分欄。
- 不在第一輪先改動 `references` 與服務頁路徑。

## 5. 舊站內容對照（摘要）

- 舊站「再生能源」偏應用場域，建議回掛至服務頁或未來應用頁。
- 舊站「工廠自動化監控系統」偏整合型總稱，建議拆入具體服務頁。
- 舊站「專案工程服務」偏交付能力，建議拆入 `I&E`、`Network Design`、`PLC / DCS`。
- 舊站「客製化系統開發」偏平台與系統能力，建議拆入 `SCADA`、`PI Server`、`Historians` 等服務頁。
