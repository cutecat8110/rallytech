# 服務架構與命名決策

最後更新：2026-03-27

## 文件目的

- 收斂服務主清單與命名策略，避免跨檔重複定義。
- 提供舊站內容對照與後續優化方向。

## 1. 決策摘要

- 第一輪服務架構承接參考站 8 項服務，不刪減核心語意。
- `一站式整合` 作為第一層獨立頁面（詳見 `02`）。
- 前台可先維持平鋪承接，3 主域分組保留為第二輪優化選項。

## 2. 服務 8 項主清單（唯一維護位置）

1. `SCADA and HMI Graphics`
2. `I&E Services`
3. `PLC & DCS Programming and Migration`
4. `PI Server`
5. `Historians`
6. `Network Design`
7. `Remote Monitoring and Data`
8. `Alarm Monitoring`

## 3. 命名策略

命名原則：

- 導覽易查找優先，其次才是完整語義
- 保留關鍵專有名詞，不為白話化而失真
- 中文與英文穩定對齊，避免跨語系結構失配

命名分層：

- `短名`：主選單、卡片、首頁模組
- `正式名`：頁面 H1、SEO title、內頁說明

## 4. 第二輪優化方向（可選）

3 個技術主域（資訊架構優化用）：

- `監控與控制`
- `資料平台`
- `儀電與網路`

用途：

- 首頁與服務總覽先顯示主域，再展開 8 項細部服務
- 主選單可用 mega menu 按主域分欄

## 5. 舊站內容對照（摘要）

- 舊站「再生能源」偏應用場域，建議回掛至服務頁或未來應用頁
- 舊站「工廠自動化監控系統」偏整合型總稱，建議拆入具體服務頁
- 舊站「專案工程服務」偏交付能力，建議拆入 `I&E`、`Network`、`PLC/DCS`
- 舊站「客製化系統開發」偏平台與系統能力，建議拆入 SCADA/PI/Historian 等服務
