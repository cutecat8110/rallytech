# 服務架構決策

最後更新：2026-03-16

## 文件目的

- 回答一個核心問題：`Pentagon` 的服務分法，是否已經是雷力新版網站最好的服務資訊架構。
- 把「能力範圍」和「前台 IA 呈現」分開判斷，避免因為同一份清單可用，就直接照搬成網站導覽。

## 0. 目前適用方式

- 本文件保留服務 IA 的評估與優化判斷。
- 但依 2026-03-16 最新規劃，第一輪網站 IA 先以參考網站結構為主。
- 也就是說：
  - `Services` 先承接參考站的 8 項服務。
  - `One Touch Experience` 另作為第一層頁面承接，中文暫定為 `一站式整合`。
  - 本文件中的 `3 個技術主域` 目前屬於後續可採用的優化方向，不是第一輪必做條件。

## 1. 先講結論

- `Pentagon` 的 8 項服務，已可直接作為雷力新版網站的底層服務架構。
- 但若把這 8 項服務直接平鋪成前台主架構，仍不是最佳解。
- 最佳做法是：
  - 保留 `Pentagon` 8 項服務，不刪減、不改掉核心語意。
  - 在前台上方再加一層技術導向的分組，讓導覽更有結構。
  - 服務頁仍維持 8 個獨立頁面，方便 SEO、雙語與後續擴充。

## 2. 為什麼 `Pentagon` 8 項服務可以保留

### 2.1 覆蓋面完整

- 就目前已確認資訊，這 8 項服務已完整覆蓋雷力對外要呈現的能力盤。
- 舊站四大服務雖然切法不同，但拆開後都能對應到這 8 項服務。

### 2.2 用語適合專業客戶

- 既然新版網站的主要受眾是專業客戶，`SCADA`、`PLC`、`DCS`、`I&E`、`PI`、`Historian` 這類詞彙本身就是有效資訊。
- 對這類受眾而言，過度白話化反而可能讓服務邊界變模糊。

### 2.3 有利於後續頁面擴充

- 8 個獨立服務頁面，對 SEO、雙語翻譯、案例掛載、技術說明與 行動按鈕配置都比較有利。
- 若未來有案例、產業應用或產品代理頁，仍可回掛到這 8 個服務頁，不需要重做整個架構。

## 3. 為什麼 `Pentagon` 的平鋪方式還不是最佳解

### 3.1 分類軸不完全一致

- `SCADA and HMI Graphics`、`PI Server`、`Historians` 比較像技術平台或系統層。
- `I&E Services`、`Network Design` 比較像工程交付或基礎設施能力。
- `Alarm Monitoring`、`Remote Monitoring and Data` 比較像運轉功能或系統能力。
- 也就是說，這 8 項服務混合了不同分類軸，適合當能力清單，但不夠適合直接當單層 IA。

### 3.2 粒度不平均

- `I&E Services` 是範圍很大的工程類別，底下可再拆很多子項。
- `Alarm Monitoring` 則相對聚焦，常會被視為 SCADA 或資料平台的一部分。
- 若全部放在同一層級，前台視覺上容易出現「有些很大、有些很細」的不均衡感。

### 3.3 專業客戶雖然懂術語，仍然需要快速掃讀

- 專業客戶不需要過度解釋基礎詞彙，但仍需要在 5 到 10 秒內判斷這家公司主要做哪幾類事。
- 因此網站仍需要一層結構化分組，讓訪客先看到能力主軸，再進入細部頁面。

## 4. 建議的最佳化版本

### 4.1 第一層

- `服務項目`

### 4.2 第二層：3 個技術主域

- 這 3 個名稱已依台灣繁中語感、專業語境、UI 掃讀性與雙語維護性綜合評估。

#### A. 監控與控制

- 對應 SCADA / HMI、PLC / DCS、告警與遠端監控。
- 服務清單：
  - `SCADA and HMI Graphics`
  - `PLC & DCS Programming and Migration`
  - `Alarm Monitoring`
  - `Remote Monitoring and Data`

#### B. 資料平台

- 對應 PI System、Historian、資料整合與歷史資料應用。
- 服務清單：
  - `PI System / PI Server`
  - `Historians`

#### C. 儀電與網路

- 對應 I&E、工業通訊網路與現場工程交付。
- 服務清單：
  - `I&E Services`
  - `Network Design`

## 5. 這個版本為什麼比直接照搬更好

- 仍然完整保留 `Pentagon` 8 項服務，沒有偏離已確認的能力盤。
- 讓首頁、服務總覽頁與主選單可以先講 3 個主域，再展開 8 項細部服務。
- 對專業客戶來說，3 個主域比 8 個平鋪標題更容易快速掃讀。
- 對內部維運來說，也更容易把案例、應用場域與解決方案內容掛到正確區塊。

## 6. 前台呈現原則

### 6.1 服務總覽頁

- 第一屏先呈現 3 個技術主域短名。
- 每個主域下方再列出對應服務卡片。
- 不建議第一屏就直接攤開 8 個沒有上下關係的平行卡片。

### 6.2 主選單

- 主選單可以只放 `Services`。
- 下拉選單或 mega menu 再用 3 個主域做欄位分組。
- 每個欄位列出 2 到 4 個具體服務。

### 6.3 服務頁標題

- 導覽、卡片與選單使用短名。
- 頁面 H1、SEO title 與內頁 heading 使用正式名。
- 不建議把正式英文名稱完全藏起來，也不建議只留英文不給中文對照。

## 7. 命名原則與建議寫法

### 7.1 綜合評估標準

- `台灣繁中語感`
  - 優先使用台灣業界較自然的詞，例如 `資料`、`儀電`、`圖控`，避免過度翻譯或偏陸系詞感。
- `專業度`
  - 保留 `SCADA`、`PLC`、`DCS`、`PI System`、`Historian` 等真正有辨識度的術語，不為了白話而失真。
- `UIUX 可視度`
  - 導覽名稱需在第一眼可掃讀，避免超過一行、避免過多連接詞，避免所有項目都長得像說明句。
- `層級辨識`
  - 主域名稱應像分類，服務名稱應像可點擊項目，兩者不能都寫得像完整句子。
- `雙語維護`
  - 中文短名要能穩定對應英文正式名，避免日後中英文頁面結構失配。

### 7.2 命名原則

- 一個名稱不要同時承擔「導覽好找」與「語意完整」兩種任務。
- 前台應採兩層命名：
  - `短名`
  - `正式名`
- `短名` 用於主選單、服務總覽卡片、首頁模組標題。
- `正式名` 用於頁面 H1、SEO title、頁首說明與雙語對照。
- 短名優先刪掉贅字，例如：
  - `服務`
  - `系統`
  - `管理`
  - `平台`
  - `設計`
- 但若刪掉後會失去辨識度，就保留關鍵詞。
- 專有名詞如 `SCADA`、`PLC`、`DCS`、`PI System`、`Historian` 可直接保留，不必硬翻成更長的中文。

### 7.3 3 個技術主域最終建議

| 用途     | 建議用字     | 說明                                                                     |
| -------- | ------------ | ------------------------------------------------------------------------ |
| 導覽短名 | `監控與控制` | 在台灣繁中語感下自然、直覺，且能同時涵蓋 SCADA / HMI 與 PLC / DCS。      |
| 導覽短名 | `資料平台`   | 雖然較抽象，但在 PI System / Historian 的脈絡中已足夠明確，UI 也最乾淨。 |
| 導覽短名 | `儀電與網路` | `儀電` 是台灣工業現場可接受的專業簡稱，搭配 `網路` 後辨識度高且不冗長。  |

### 7.4 8 項服務最終建議

| Pentagon 對應                         | 導覽短名      | 正式名                               |
| ------------------------------------- | ------------- | ------------------------------------ | ------------------------- |
| `SCADA and HMI Graphics`              | `SCADA / HMI` | `SCADA and HMI Graphics              | SCADA / HMI 圖控`         |
| `PLC & DCS Programming and Migration` | `PLC / DCS`   | `PLC & DCS Programming and Migration | PLC / DCS 程式開發與移轉` |
| `Alarm Monitoring`                    | `告警監控`    | `Alarm Monitoring                    | 告警監控`                 |
| `Remote Monitoring and Data`          | `遠端監控`    | `Remote Monitoring and Data          | 遠端監控與資料管理`       |
| `PI System / PI Server`               | `PI System`   | `PI System / PI Server               | PI 資料平台`              |
| `Historians`                          | `歷史資料庫`  | `Historians                          | Historian 歷史資料庫`     |
| `I&E Services`                        | `儀電工程`    | `I&E Services                        | 儀電工程`                 |
| `Network Design`                      | `工業網路`    | `Network Design                      | 工業網路設計`             |

### 7.5 命名調整的理由

- `SCADA / HMI` 已足夠讓專業客戶理解，不必在選單層就加上 `圖控`。
- `PLC / DCS` 在導覽層可直接成立，`程式開發與移轉` 留到正式名即可。
- `遠端監控` 在查找上比 `遠端監控與資料管理` 快，資料面留在正式名補足。
- `PI System` 屬高辨識度專有名詞，短名保留英文較準確。
- `歷史資料庫` 比單用 `Historian` 更符合中文介面掃讀習慣，也更符合台灣常見說法。
- `儀電工程` 比 `儀控與電氣工程服務` 更俐落，也更像實務用語。
- `工業網路` 比 `工業通訊網路設計` 更適合選單，但正式名仍建議保留 `設計`。

## 8. 重要服務如何呈現

- 若網站需要先講「重要服務」，建議以 3 個技術主域作為首頁第一層重點，而不是先替 8 個服務做主次排序。
- 原因是：
  - 3 個主域能代表雷力的能力結構。
  - 8 個服務則代表可對外拆開說明的細部項目。
  - 這樣既不會弱化任何既有能力，也能讓訪客快速理解服務版圖。

## 9. 研究註記

- 依 2026-03-16 查閱公開網站，`Pentagon` 的 `/services/` 頁面對外列出 8 項服務。
- 同一網站其他頁面仍可看到更細的工程能力描述，代表這 8 項服務本質上更接近「對外主服務分類」，而不是完整能力細目清單。
- 這也進一步支持本文件的判斷：8 項服務應保留，但前台 IA 最好加上結構化分組。

## 10. 決策

- 定案方向：
  - 第一輪 IA 先承接 `Pentagon` 的 8 項服務與 `One Touch Experience`。
  - `One Touch Experience` 在雷力網站中暫定中文名稱為 `一站式整合`。
  - `3 個技術主域` 與 `短名 + 正式名` 的做法保留為下一輪優化選項。
  - 中文命名仍以專業、清楚、可維持雙語一致性為前提。
