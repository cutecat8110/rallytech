# 專案基線

最後更新：2026-03-16

## 文件目的

- 定義目前已確認的專案前提，作為後續資訊架構、文案與技術決策的共同基線。
- 避免未來在不同草稿之間重複確認相同問題。

## 1. 專案定位

- 專案類型：企業官網重構。
- 專案目的：重新整理雷力的服務範圍、能力結構與對外呈現方式，讓網站能清楚支撐品牌信任、業務溝通與聯絡轉換。
- 本階段重點：先把資訊架構、服務分類、內容策略與維運前提定清楚，再進入設計與開發。

## 2. 主要受眾

- 主要受眾不是一般消費者，也不是完全沒有技術背景的訪客。
- 主要對象應視為具備工程、製程、自動化、儀控、廠務或專案管理背景的專業客戶。
- 網站應優先服務下列角色：
  - 業主端工程或廠務窗口
  - 自動化 / 儀控 / 電氣相關主管或工程師
  - 專案管理、採購或系統整合決策者

## 3. 溝通原則

- 服務命名可保留 `SCADA`、`HMI`、`PLC`、`DCS`、`PI System`、`Historian`、`I&E` 等專業用語。
- 中文頁面不必過度白話化，但必須讓專業客戶能快速看出服務範圍、交付內容與能力邊界。
- 命名應分成兩層：
  - `短名` 給導覽、卡片與查找
  - `正式名` 給頁面標題、SEO 與雙語對照
- 命名優先順序如下：
  - 先好找
  - 再精準
  - 再一致
  - 最後才是白話
- 中文與英文應採並列策略，而不是只保留其中一種。
- 首頁與服務總覽頁應先講清楚「我們做哪些服務」與「哪些是核心能力」，而不是先講抽象品牌敘事。

## 4. 已確認範圍

- 首波網站至少包含：
  - 首頁
  - 關於我們
  - 服務項目
  - 一站式整合
  - 聯絡我們
  - 中文 / 英文雙語
  - 後台查看聯絡表單資料
- 首波不做但需預留：
  - 代理產品介紹
  - 詢價車
  - 部落格
  - 人才招募

## 5. 服務架構基線

- 現階段內容與分類以 `Pentagon` 參考網站為主，採 `先完整承接，再逐步調整`。
- 雷力的服務範圍已確認與 `Pentagon` 公開的 8 項服務一致。
- 這 8 項服務目前為：
  - `SCADA and HMI Graphics`
  - `I&E Services`
  - `PLC & DCS Programming and Migration`
  - `PI Server`
  - `Historians`
  - `Network Design`
  - `Remote Monitoring and Data`
  - `Alarm Monitoring`
- `One Touch Experience` 已納入新版第一層 IA，中文暫定為 `一站式整合`。
- 雷力舊官網內容目前保留作歷史背景資料，暫不作為第一輪 IA 與內容規劃的主要來源。

## 6. 新站要解決的核心問題

- 目前需要的新站不是延續舊站分類，而是先建立一套可直接運作的新版 IA。
- 第一輪以參考網站為主，可先快速確立頁面邊界、內容骨架與雙語對照。
- 後續再評估哪些內容需要在地化、重組或重新命名。

## 7. 轉換與信任目標

- 新站應支撐的主要轉換行為：
  - 聯絡諮詢
  - 專案初步洽談
  - 能力資格確認
- 新站應逐步補強的信任訊號：
  - 服務範圍與技術深度
  - 整合方案與交付能力
  - 專案交付能力
  - 安全衛生與品質相關承諾
  - 聯絡回應效率與窗口清楚度

## 8. 後台方向

- 首波後台定位為輕量管理後台，不預設一開始就做完整 CMS。
- 首波至少應支援：
  - 聯絡表單列表
  - 單筆資料查看
  - 姓名、公司、Email、電話、需求內容、送出時間等基本欄位
- 後續可預留：
  - 狀態管理
  - 搜尋與篩選
  - 匯出資料
  - 權限與帳號管理
  - 未來其他模組共用資料結構

## 9. 目前待決策項目

- 前台 `Services` 第一輪是否維持 8 項平鋪，或先做第二層分組。
- `一站式整合` 底下 5 個項目的中文命名方式。
- 服務頁標題要採「英文在前」或「中文在前」。
- 首頁要如何排序核心服務、一站式整合與信任訊號。
- 雙語內容未來由誰維護，以及是否需要後台管理語系內容。

## 10. 前端實作共識（新增）

- 「Nuxt UI First」為核心宗旨：互動元件優先採 Nuxt UI，不為了短期方便手刻替代元件。
- 預設以繁體中文作為協作與頁面初稿文案語言；英文版採並列補齊，不以英文單語先行。
- 協作節奏採「先討論目標與細節，再實作」，且以小步迭代方式逐段完成，不一次產出整站。
- 實作時不主動新增未確認區塊，僅處理已確認範圍，避免產生多餘內容。
- Vue 頁面與元件開發時，`<script setup lang="ts">` 與 `<template>` 皆採中文註釋，重點描述區塊目的與行為意圖。
- 註釋需描述「目的與意圖」，避免只是重複程式碼字面意思。
- 頁面層以組裝區塊為主，區塊內容優先就地放在元件內，不先集中在父層傳遞。
- `app/content` 預設不使用於少量頁面文案；僅在高複用且確有必要時才評估抽離。
- 型別策略以推論優先，元件內就近定義為主；跨模組共用或 API 契約才放入 `app/types`。
- 主要行動按鈕（白字）預設採 `bg-primary-700 hover:bg-primary-600 active:bg-primary-800`，避免使用 Primary 搭配深色文字作為主要行動按鈕。
- `UButton` 若使用 `color="primary" + variant="solid"`，文字需維持白字；禁止覆蓋為深色文字。
- 樣式以語意 class 與設計 token 為主，避免依賴 `!important`（含 Tailwind `!` 前綴）覆寫。
- 禁止使用高優先級全域連結字色 reset（例如 `a { color: inherit; }`），避免覆蓋 Nuxt UI 按鈕字色。
- 若 Nuxt UI 無法滿足必要行為才可自製 UI，且需在元件用中文註釋說明原因與限制。
- 若遇到第三方樣式衝突，需先調整層級與結構；只有在必要時才可使用 `!important`，並附註解說明原因。
- 一般開發流程預設不需執行 `npm run build`；以 `npm run lint` 與 `npm run typecheck` 為主要驗證。
- 若需要比對參考網站與本機頁面實際渲染結果，優先使用 `chrome-devtools-mcp` 做實查，而非只靠程式碼推測。
- 本專案已成功驗證 `chrome-devtools-mcp` 可觀察 `https://www.rallytech.com.tw/` 與本機開發站 `http://127.0.0.1:3000/`。
- 本機驗證建議流程：先啟動 `npx nuxt dev --host 127.0.0.1 --port 3000`，再用 `chrome-devtools` 工具檢查載入、可見文字、視覺特徵與語意結構。
- 若要穩定驗證工具可用性，優先採一次性 `--isolated --headless` 測試；若要附著到使用者既有 Chrome，再使用 `autoConnect`。
