# 專案策略與基線總綱

最後更新：2026-04-02

## 文件目的

- 定義專案共同前提與策略邊界。
- 提供後續 IA、設計、文案與開發的單一對齊基準。

## 適用角色

- 管理、業務、老闆：確認這一輪官網重建到底要做到哪裡。
- 設計、前端、PM：確認共同原則、頁面範圍與交付邊界。
- AI agent：作為第一份策略基線，不在實作時擴張到未確認範圍。

## 1. 專案定位

- 專案類型：雷力科技企業官網重構
- 核心目的：清楚呈現服務能力、建立信任、提升聯絡轉換
- 執行節奏：先建立可運作的資訊與內容骨架，再進入視覺變體與程式落地

## 2. 主要受眾

- 工程、製程、自動化、儀控、廠務、專案管理等專業客戶
- 主要角色：
  - 業主端工程或廠務窗口
  - 自動化 / 儀控 / 電氣主管與工程師
  - 專案管理、採購或系統整合決策者

## 3. 溝通與語言策略

- 文案語氣：專業、清楚、可信賴，避免過度口語化與空泛行銷語氣
- 專業術語可保留：`SCADA`、`HMI`、`PLC`、`DCS`、`PI System`、`Historian`、`I&E`
- 命名採雙層：
  - `短名`：導覽、卡片、查找
  - `正式名`：頁面標題、SEO、雙語對照
- 語言策略：繁體中文優先，英文並列維護

## 4. 目前共識（官網重建）

- 工作方向：先精準承接參考站可見內容與頁面骨架，再依我方設計系統做視覺變體與互動優化。
- 視覺策略：不直接照抄配色；版型與資訊層級可承接，但色彩、字體、元件語氣與品牌表現要回到雷力科技設計系統。
- 內容策略：`source -> en -> zh-tw` 三層同步維護，`source` 保留參考站可見原文，`en` 與 `zh-tw` 改成我方公司稿與台灣 B2B 官網語感。
- references 策略：以所見及所得方式記錄區塊、文字與必要註記，供管理、業務、設計與工程共同對照。
- 圖片策略：優先使用可追溯的開源圖片來源，不直接以人工另存作為正式流程。
- 互動策略：需納入滾動後顯示內容、hover、accordion、tab、form 與 desktop/mobile 驗證，不只比對靜態畫面。

## 5. 範圍邊界

首波必做：

- 首頁、關於我們、服務總覽、8 項服務內頁、一站式整合、聯絡我們、FAQ 輔助頁
- 中英文雙語
- 聯絡表單資料的後台基本查看

首波不做但需預留：

- 代理產品介紹
- 詢價車或報價流程
- 部落格 / 新聞 / 案例資料庫
- 人才招募專區

## 6. 轉換與信任目標

主要轉換行為：

- 聯絡諮詢
- 專案初步洽談
- 能力資格確認

核心信任訊號：

- 服務深度與交付能力
- 整合方案可行性
- 安全與品質承諾
- 聯絡窗口與回應效率

## 7. 工具與執行邊界

- 本機核心 MCP baseline：`chrome-devtools` + `playwright` + `context7` + `fetch`
- 本機 enabled optional MCP：`memory`
- `github` 目前不在本機 config；若 session 或 connector 提供，視為 review / trace integration，而不是本機 baseline
- 核心設計 / 內容 skills：`uiux-pm-collaboration`、`frontend-skill`、`document-writer`
- 視覺 / 文案 / 資產輔助 skills：`visual-qa-review`、`b2b-website-copy-strategy`、`asset-sourcing-licensing`
- Nuxt 落地 skills：`nuxt`、`nuxt-ui`、`vue`、`nuxt-seo`
- 工具原則：不採用需付費的 crawler 或 browser automation 服務作為正式基線
- `js_repl = true` 代表 workstation 已準備好互動式 debug 能力，但是否可用仍需看當前 session
- clone Phase 1 邊界：先做首頁與代表性頁面的高擬真重建，不把滾動動畫列為第一輪驗收項目

## 8. 後台方向

- 首波定位：輕量管理後台，不預設完整 CMS
- 最低需求：
  - 聯絡表單列表
  - 單筆查看（姓名、公司、Email、電話、需求、時間）
- 後續擴充：
  - 狀態
  - 搜尋與篩選
  - 匯出
  - 權限
  - 共用資料結構
